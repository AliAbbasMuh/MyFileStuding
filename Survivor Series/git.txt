Setting username
يتم استخدام اسم المستخدم بواسطة Git لكل عملية تنفيذ.
$ git config --global user.name "Ali Abbas"

Setting email id
يستخدم Git معرف البريد الإلكتروني هذا لكل التزام.
$ git config --global user.email  "aardwolf.xxvll@gmail.com"

Setting editor
محرر الإعداد
يمكنك تعيين محرر النص الافتراضي عندما يطلب منك Git كتابة رسالة. إذا لم تحدد أيًا من المحررين ، فسيستخدم Git محرر النظام الافتراضي الخاص بك.
لتحديد محرر نصوص مختلف ، مثل Vim ،
$ git config --global core.editor Vim

Checking Your Settings
التحقق من الإعدادات الخاصة بك
يمكنك التحقق من إعدادات التكوين الخاصة بك ؛ يمكنك استخدام الأمر git config --list لسرد جميع الإعدادات التي يمكن أن يجدها Git في تلك المرحلة.
$ git config --list

Colored output
الإخراج الملون
يمكنك تخصيص إخراج Git لعرض سمة ألوان مخصصة. يمكن استخدام تكوين git لتعيين سمات الألوان هذه.
Color.ui
$ Git config --global color.ui true

مستويات التكوين Git
يمكن للأمر git config قبول الوسائط لتحديد مستوى التكوين. تتوفر مستويات التكوين التالية في تكوين Git.
محلي

--محلي
إنه المستوى الافتراضي في Git. سيكتب Git config على المستوى المحلي إذا لم يتم توفير خيار التكوين. يتم تخزين قيم التكوين المحلية في دليل .git / config كملف.
--عالمي
تكوين المستوى العام هو تكوين خاص بالمستخدم. وسائل خاصة بالمستخدم ، يتم تطبيقها على مستخدم نظام تشغيل فردي. يتم تخزين قيم التكوين العامة في الدليل الرئيسي للمستخدم. ~ /.gitconfig في أنظمة UNIX و C: \ Users \\. gitconfig على windows كتنسيق ملف.
--النظام
يتم تطبيق التكوين على مستوى النظام عبر نظام بأكمله. النظام بأكمله يعني جميع المستخدمين على نظام التشغيل وجميع المستودعات. يُخزن ملف التكوين على مستوى النظام في ملف gitconfig خارج دليل النظام. $ (بادئة) / etc / gitconfig في أنظمة UNIX و C: \ ProgramData \ Git \ config على Windows.

ترتيب أولوية تكوين Git محلي وعالمي ونظام ، على التوالي. هذا يعني أنه عند البحث عن قيمة التكوين ، سيبدأ Git على المستوى المحلي وينتقل إلى مستوى النظام.

Git Tools
أدوات Git
لاستكشاف الوظائف القوية لـ Git ، نحتاج إلى بعض الأدوات. يأتي Git مع بعض أدواته مثل Git Bash و Git GUI لتوفير الواجهة بين الجهاز والمستخدم. وهو يدعم أدوات داخلية وكذلك أدوات الطرف الثالث.

يأتي Git مع أدوات واجهة المستخدم الرسومية المضمنة مثل git bash و git-gui و gitk للالتزام والتصفح. كما أنه يدعم العديد من أدوات الجهات الخارجية للمستخدمين الذين يبحثون عن تجربة خاصة بالنظام الأساسي.

Git Package Tools
أدوات حزمة Git
يوفر Git وظائف قوية لاستكشافه. نحتاج إلى العديد من الأدوات مثل الأوامر وسطر الأوامر و Git GUI. دعونا نفهم بعض أدوات الحزم الأساسية.

GitBash
Git Bash هو تطبيق لبيئة Windows. يتم استخدامه كسطر أوامر Git للنوافذ. يوفر Git Bash طبقة محاكاة لتجربة سطر أوامر Git. Bash هو اختصار لـ Bourne Again Shell. يحتوي مثبت حزمة Git على Bash ، وأدوات مساعدة bash ، و Git على نظام تشغيل Windows.

Bash هو قشرة افتراضية قياسية في Linux و macOS. القشرة هي تطبيق طرفي يستخدم لإنشاء واجهة مع نظام تشغيل من خلال الأوامر.

بشكل افتراضي ، تحتوي حزمة Git Windows على أداة Git Bash. يمكننا الوصول إليه عن طريق النقر بزر الماوس الأيمن على مجلد في مستكشف Windows.

Git Bash Commands
أوامر جيت باش
يأتي Git Bash مع بعض الأوامر الإضافية المخزنة في دليل / usr / bin لمحاكاة Git Bash. يمكن أن يوفر Git Bash تجربة صدفة قوية على Windows. يأتي Git Bash مع بعض أوامر shell الأساسية مثل Ssh و scp و cat و find.

يتضمن Git Bash أيضًا المجموعة الكاملة من أوامر Git core مثل git clone و git الالتزام و git checkout و git push والمزيد.

Git GUI
بوابة GUI
Git GUI هو بديل قوي لـ Git BASH. يوفر إصدارًا رسوميًا لوظيفة سطر أوامر Git ، بالإضافة إلى أدوات فرق مرئية شاملة. يمكننا الوصول إليه بمجرد النقر بزر الماوس الأيمن على مجلد أو موقع في مستكشف Windows. أيضًا ، يمكننا الوصول إليه من خلال سطر الأوامر عن طريق كتابة الأمر أدناه.

$ git gui
يسهل Git استخدام بعض أدوات واجهة المستخدم الرسومية المضمنة للالتزام (git-gui) والتصفح (gitk) ، ولكن هناك العديد من أدوات الجهات الخارجية للمستخدمين الذين يبحثون عن تجربة خاصة بالنظام الأساسي.

Git Terminology
مصطلحات Git
Git هي أداة غطت العديد من المصطلحات والمصطلحات ، والتي غالبًا ما تكون صعبة للمستخدمين الجدد ، أو أولئك الذين يعرفون أساسيات Git ولكنهم يريدون أن يصبحوا خبراء في Git. لذا ، نحن بحاجة إلى شرح بسيط للمصطلحات الكامنة وراء الأدوات. دعنا نلقي نظرة على المصطلحات شائعة الاستخدام.

بعض المصطلحات شائعة الاستخدام هي:
Branch
فرع
الفرع هو نسخة من المستودع تختلف عن مشروع العمل الرئيسي. إنها ميزة أساسية متوفرة في معظم أنظمة التحكم في الإصدارات الحديثة. يمكن أن يحتوي مشروع Git على أكثر من فرع. يمكننا إجراء العديد من العمليات على فرع Git مثل إعادة التسمية والقائمة والحذف وما إلى ذلك.

Checkout
الدفع
في Git ، يتم استخدام مصطلح checkout لإجراء التبديل بين الإصدارات المختلفة للكيان المستهدف. يتم استخدام الأمر git checkout للتبديل بين الفروع في المستودع.

Cherry-Picking
قطف الكرز
يُقصد بقطف الكرز في Git تطبيق بعض الالتزامات من فرع إلى فرع آخر. في حال ارتكبت خطأ وقمت بتغيير الفرع الخطأ ، ولكن لا تريد دمج الفرع بأكمله. يمكنك التراجع عن الالتزام واختياره من فرع آخر.

Clone
استنساخ
git clone هو أداة سطر أوامر Git. يتم استخدامه لعمل نسخة من المستودع الهدف أو استنساخه. إذا كنت أرغب في الحصول على نسخة محلية من مستودعي من GitHub ، فإن هذه الأداة تسمح بإنشاء نسخة محلية من هذا المستودع على دليلك المحلي من عنوان URL الخاص بالمستودع.
Fetch
أحضر
يتم استخدامه لجلب الفروع والعلامات من واحد أو أكثر من المستودعات الأخرى ، جنبًا إلى جنب مع الكائنات الضرورية لإكمال تواريخها. يقوم بتحديث فروع التتبع عن بعد.
HEAD
رأس
HEAD هو تمثيل آخر التزام في فرع السداد الحالي. يمكننا التفكير في الرأس مثل الفرع الحالي. عند تبديل الفروع باستخدام git checkout ، تتغير مراجعة HEAD وتوجه الفرع الجديد.
Index
فهرس
فهرس Git هو منطقة انطلاق بين دليل العمل والمستودع. يتم استخدامه كفهرس لإنشاء مجموعة من التغييرات التي تريد الالتزام بها معًا.
Master
رئيس
Master هو اصطلاح تسمية لفرع Git. إنه فرع افتراضي لـ Git. بعد استنساخ مشروع من خادم بعيد ، يحتوي المستودع المحلي الناتج على فرع محلي واحد فقط. هذا الفرع يسمى الفرع "الرئيسي". وهذا يعني أن "الرئيسي" هو الفرع "الافتراضي" للمستودع.
Merge
دمج
الدمج هو عملية لإعادة التاريخ المتشعب معًا. يسهل عليك الأمر git merge أخذ البيانات التي تم إنشاؤها بواسطة فرع git ودمجها في فرع واحد.
Origin
أصل
في Git ، "الأصل" هو إشارة إلى المستودع البعيد من مشروع تم استنساخه في البداية. بتعبير أدق ، يتم استخدامه بدلاً من عنوان URL للمستودع الأصلي لتسهيل الرجوع إليه.
Pull/Pull Request
طلب سحب / سحب
يستخدم المصطلح Pull لتلقي البيانات من GitHub. يقوم بجلب التغييرات ودمجها على الخادم البعيد في دليل العمل الخاص بك. يتم استخدام الأمر git pull لعمل سحب Git.

طلبات السحب هي عملية للمطور لإعلام أعضاء الفريق بأنهم أكملوا ميزة. بمجرد أن يصبح فرع الميزات جاهزًا ، يقدم المطور طلب سحب عبر حساب الخادم البعيد. يعلن طلب السحب عن جميع أعضاء الفريق الذين يحتاجون إلى مراجعة الكود ودمجه في الفرع الرئيسي.
Push
يدفع
يشير مصطلح الدفع إلى تحميل محتوى مستودع محلي إلى مستودع بعيد. الدفع هو عملية نقل تلتزم بها من مستودعك المحلي إلى مستودع بعيد. الدفع قادر على الكتابة فوق التغييرات ؛ يجب توخي الحذر عند الدفع.
Rebase
ريباس
في Git ، يُشار إلى مصطلح rebase على أنه عملية نقل أو دمج سلسلة من الالتزامات في التزام أساسي جديد. تعد إعادة التأسيس مفيدة للغاية ويتم تصور العملية في بيئة سير عمل تفريع الميزة.

من منظور المحتوى ، تعد إعادة التأسيس أسلوبًا لتغيير قاعدة الفرع الخاص بك من التزام إلى آخر.
Remote
بعيد
في Git ، يتعلق المصطلح عن بعد بالمستودع البعيد. إنه مستودع مشترك يستخدمه جميع أعضاء الفريق لتبادل تغييراتهم. يتم تخزين المستودع البعيد في خدمة استضافة التعليمات البرمجية مثل الخادم الداخلي و GitHub و Subversion والمزيد.

في حالة وجود مستودع محلي ، لا يوفر جهاز التحكم عن بُعد عادةً شجرة ملفات للحالة الحالية للمشروع ، وكبديل لذلك فهو يتكون فقط من بيانات تعيين الإصدار .git.
Repository
مخزن
في Git ، يشبه Repository بنية البيانات التي يستخدمها VCS لتخزين البيانات الوصفية لمجموعة من الملفات والأدلة. يحتوي على مجموعة الملف بالإضافة إلى محفوظات التغييرات التي تم إجراؤها على هذه الملفات. تعتبر المستودعات في Git بمثابة مجلد مشروعك. يحتوي المستودع على جميع البيانات المتعلقة بالمشروع. المشاريع المتميزة لها مستودعات متميزة.
Stashing
يخفي
في بعض الأحيان تريد تبديل الفروع ، لكنك تعمل على جزء غير مكتمل من مشروعك الحالي. أنت لا تريد أن تلتزم بعمل نصف مكتمل. يسمح لك Git Stashing بالقيام بذلك. يمكّنك الأمر git stash من تبديل الفرع دون تنفيذ الفرع الحالي.
Tag
بطاقة شعار
تشير العلامات إلى نقطة معينة في تاريخ Git. يتم استخدامه لتمييز مرحلة الالتزام على أنها مهمة. يمكننا وضع علامة على الالتزام للرجوع إليه في المستقبل. في المقام الأول ، يتم استخدامه لتمييز نقطة أولية للمشروع مثل v1.1. هناك نوعان من العلامات.
Light-weighted tag
علامة خفيفة الوزن
علامة مشروحة
المنبع والمصب
مصطلح المنبع والمصب هو مرجع للمستودع. بشكل عام ، المنبع هو المكان الذي نسخت فيه المستودع من (الأصل) والمصب هو أي مشروع يدمج عملك مع أعمال أخرى. ومع ذلك ، لا تقتصر هذه الشروط على مستودعات Git.
Git Revert
بوابة العودة
في Git ، يتم استخدام مصطلح العودة لعكس بعض الالتزام. للعودة إلى الالتزام ، يتم استخدام الأمر git revert. إنه أمر نوع التراجع. ومع ذلك ، فهو ليس بديلاً تقليديًا للتراجع.
Git Reset
إعادة تعيين بوابة
في Git ، يشير مصطلح إعادة التعيين إلى التراجع عن التغييرات. يتم استخدام الأمر git reset لإعادة تعيين التغييرات. يحتوي الأمر git reset على ثلاثة أشكال أساسية من الاستدعاء. هذه الأشكال على النحو التالي.

لين
مختلط
صعب
جيت تجاهل
في Git ، يستخدم مصطلح التجاهل لتحديد الملفات التي لم يتم تعقبها عن قصد والتي يجب أن يتجاهلها Git. لا يؤثر على الملفات التي تم تعقبها بالفعل بواسطة Git.

Git Diff
بوابة الفرق
Git diff هو أداة مساعدة لسطر الأوامر. إنه أمر Git متعدد الاستخدامات. عند تنفيذه ، يتم تشغيل دالة فرق على مصادر بيانات Git. يمكن أن تكون مصادر البيانات هذه ملفات وفروع وتعهدات والمزيد. يتم استخدامه لإظهار التغييرات بين الالتزامات والالتزامات وشجرة العمل ، إلخ.

Git Cheat Sheet
ورقة الغش بوابة
ورقة الغش في Git هي ملخص لمراجع Git السريعة. يحتوي على أوامر Git الأساسية مع التثبيت السريع. ورقة الغش أو ورقة السرير هي مجموعة مختصرة من الملاحظات المستخدمة كمرجع سريع. تم تسمية أوراق الغش بهذا الاسم لأن الناس قد يستخدمونها دون معرفة مسبقة.

Git Flow
تدفق بوابة
GitFlow هو نموذج متفرع لـ Git ، تم تطويره بواسطة Vincent Driessen. إنه منظم بشكل جيد للغاية للتعاون وتوسيع نطاق فريق التطوير. Git flow عبارة عن مجموعة من أوامر Git. ينجز العديد من عمليات المستودع بأوامر واحدة فقط.

Git Squash
جيت سكواش
في Git ، يستخدم مصطلح الاسكواش لسحق الالتزامات السابقة في واحد. Git squash هي تقنية ممتازة للتغييرات الخاصة بالمجموعة قبل إرسالها إلى الآخرين. يمكنك دمج العديد من الالتزامات في التزام واحد باستخدام أمر rebase التفاعلي الفعال.

Git Rm
جيت Rm
في Git ، يشير المصطلح rm إلى إزالة. يتم استخدامه لإزالة الملفات الفردية أو مجموعة من الملفات. تتمثل الوظيفة الرئيسية لـ git rm في إزالة الملفات المتعقبة من فهرس Git. بالإضافة إلى ذلك ، يمكن استخدامه لإزالة الملفات من دليل العمل والفهرس المرحلي.

Git Fork
جيت فورك
الشوكة هي نسخة تقريبية من المستودع. يتيح لك إنشاء مستودع تخزين اختبار التغييرات وتصحيحها بحرية دون التأثير على المشروع الأصلي.

استخدام رائع لاستخدام الشوكات لاقتراح تغييرات لإصلاحات الأخطاء. لحل مشكلة الخطأ الذي وجدته ، يمكنك:

الفرع الرئيسي
الفرع الرئيسي هو الفرع الرئيسي للمشروع الذي يحتوي على كل تاريخ التغييرات النهائية. يجب استخدام كل مطور للفرع الرئيسي. يحتوي الفرع الرئيسي على الكود المصدري لـ HEAD الذي يعكس دائمًا نسخة نهائية من المشروع.
يحتوي المستودع المحلي الخاص بك على الفرع الرئيسي الخاص به والذي يتم تحديثه دائمًا مع الرئيسي للمستودع البعيد.

يقترح عدم العبث مع السيد. إذا قمت بتحرير الفرع الرئيسي لمشروع جماعي ، فستؤثر تغييراتك على أي شخص آخر ، وبسرعة كبيرة ، ستكون هناك تعارضات في الدمج.

Develop Branch
تطوير الفرع
إنه مواز للفرع الرئيسي. كما يعتبر الفرع الرئيسي للمشروع. يحتوي هذا الفرع على أحدث تغييرات التطوير التي تم تسليمها للإصدار التالي. يحتوي على شفرة المصدر النهائية للإصدار. ويسمى أيضًا باسم "فرع التكامل".
عندما يصل فرع التطوير إلى نقطة مستقرة ويكون جاهزًا للإصدار ، يجب دمجه مع الرئيسي ووضع علامة عليه بنسخة إصدار.

Supportive Branches
الفروع الداعمة
يحتاج نموذج التطوير إلى مجموعة متنوعة من الفروع الداعمة للتطوير الموازي ، وتتبع الميزات ، والمساعدة في التثبيت والإصدار السريع ، وغيرها من المشكلات. هذه الفروع لها عمر محدود ويتم إزالتها بعد الاستخدامات.
الأنواع المختلفة من الفروع الداعمة التي قد نستخدمها هي كالتالي:

Basic Git Commands

$ git config --global user.email "aardwolf.xxvll@gmail.com"

أمر Git clone

Git status command
أمر Git status
يتم استخدام أمر الحالة لعرض حالة دليل العمل ومنطقة التدريج. يسمح لك بمعرفة التغييرات التي تم إجراؤها على مراحل ، والتي لم يتم تنفيذها ، وأي الملفات لم يتم تعقبها بواسطة Git. لا تظهر لك أي معلومات حول تاريخ المشروع الملتزم به. لهذا ، تحتاج إلى استخدام سجل git. يسرد أيضًا الملفات التي قمت بتغييرها وتلك التي ما زلت بحاجة إلى إضافتها أو الالتزام بها.
$ git status

Git push Command
أمر Git push
يتم استخدامه لتحميل محتوى المستودع المحلي إلى مستودع بعيد. الدفع هو عملية نقل تلتزم بها من مستودعك المحلي إلى مستودع بعيد. إنها تكملة git fetch ، ولكن في حين أن جلب الواردات يلتزم بالفروع المحلية على دفع الصادرات نسبيًا إلى الفروع البعيدة. يتم تكوين الفروع البعيدة باستخدام الأمر git remote. الضغط قادر على الكتابة فوق التغييرات ، ويجب توخي الحذر عند الدفع.
يمكن استخدام أمر Git push على النحو التالي.
بوابة دفع سيد الأصل Git push origin master
يرسل هذا الأمر التغييرات التي تم إجراؤها على الفرع الرئيسي ، إلى مستودعك البعيد.
$ git push [variable name] master
-($ git push origin master)

====
Git Add
يتم استخدام الأمر git add لإضافة محتويات ملف إلى الفهرس (منطقة التدريج) ، يقوم هذا الأمر بتحديث المحتوى الحالي لشجرة العمل إلى منطقة التدريج. كما تقوم بإعداد المحتوى المرحلي للالتزام التالي. في كل مرة نقوم فيها بإضافة أو تحديث أي ملف في مشروعنا ، يلزم إعادة توجيه التحديثات إلى منطقة التدريج.
يعد الأمر git add جزءًا أساسيًا من تقنية Git. عادةً ما يضيف ملفًا واحدًا في كل مرة ، ولكن هناك بعض الخيارات المتاحة التي يمكن أن تضيف أكثر من ملف في وقت واحد.
يحتوي "الفهرس" على لقطة لبيانات شجرة العمل. ستتم إعادة توجيه هذه اللقطة للالتزام التالي.
يمكن تشغيل الأمر git add عدة مرات قبل إجراء الالتزام. يمكن وضع جميع عمليات الإضافة هذه ضمن التزام واحد. يضيف الأمر add الملفات المحددة في سطر الأوامر.
لا يضيف الأمر git add ملف .gitignore افتراضيًا. في الواقع ، يمكننا تجاهل الملفات بهذا الأمر.
دعونا نفهم كيفية إضافة الملفات على Git؟

Git add files
بوابة إضافة الملفات
الأمر Git add هو أمر مباشر للأمام. يضيف الملفات إلى منطقة التدريج. يمكننا إضافة ملفات فردية أو متعددة مرة واحدة في منطقة التدريج. سيتم تشغيله على النحو التالي:

$ touch newfile2.txt
$ git add newfile2.txt
$ git status

Git Add All
بوابة إضافة الكل
يمكننا إضافة أكثر من ملف واحد في Git ، لكن علينا تشغيل الأمر add بشكل متكرر. يسهل لنا Git من خلال خيار فريد لأمر add يمكننا من خلاله إضافة جميع الملفات المتاحة مرة واحدة. لإضافة جميع الملفات من المستودع ، قم بتشغيل الأمر add مع الخيار -A. يمكننا ان نستخدم '.' بدلا من الخيار -A. سيقوم هذا الأمر بترتيب جميع الملفات في وقت واحد. سيتم تشغيله على النحو التالي:

$ git add -A
or
$ git add .

Add all New and Updated Files Only:
إضافة كافة الملفات الجديدة والمحدثة فقط:
يتيح لنا Git عرض الملفات المحدثة والمُنشأة حديثًا فقط مرة واحدة. سنستخدم خيار تجاهل الإزالة للقيام بذلك. سيتم استخدامه على النحو التالي:

Add all Modified and Deleted Files
أضف جميع الملفات المعدلة والمحذوفة
يسهل علينا Git add مجموعة متنوعة من الخيارات. هناك خيار آخر متاح في Git ، والذي يسمح لنا بترتيب الملفات المعدلة والمحذوفة فقط. لن يقوم بتجهيز الملف الذي تم إنشاؤه حديثًا. لتنظيم جميع الملفات المعدلة والمحذوفة فقط ، قم بتشغيل الأمر التالي:
$ git add -u

Add Files by Wildcard
إضافة الملفات عن طريق Wildcard
يتيح لنا Git إضافة جميع ملفات الأنماط نفسها مرة واحدة. إنها طريقة أخرى لإضافة ملفات متعددة معًا. لنفترض أنني أريد إضافة جميع ملفات جافا أو الملفات النصية ، ثم يمكننا استخدام النمط .java أو .txt. للقيام بذلك ، سنقوم بتشغيل الأمر على النحو التالي:
$ git add *.java
سيقوم الأمر أعلاه بترتيب جميع ملفات Java. سيتم تطبيق نفس النمط على الملفات النصية.
الخطوة التالية بعد إضافة الملفات هي الالتزام بمشاركتها على Git.

Git Undo Add
Git Undo Add
يمكننا التراجع عن عملية git add. ومع ذلك ، فهو ليس جزءًا من الأمر git add ، ولكن يمكننا القيام بذلك من خلال الأمر git reset.
للتراجع عن عملية إضافة ، قم بتشغيل الأمر التالي:
$ git reset <filename>

Git Commit
بوابة الالتزام
يتم استخدامه لتسجيل التغييرات في المستودع. إنه الأمر التالي بعد إضافة git. يحتوي كل التزام على بيانات الفهرس ورسالة الالتزام. كل التزام يشكل علاقة بين الوالدين والطفل. عندما نضيف ملفًا في Git ، فسيتم ذلك في منطقة التدريج. يتم استخدام أمر الالتزام لجلب التحديثات من منطقة التدريج إلى المستودع.
يرتبط التدريج والالتزام ببعضهما البعض. يتيح لنا التدريج الاستمرار في إجراء التغييرات على المستودع ، وعندما نريد مشاركة هذه التغييرات على نظام التحكم في الإصدار ، يسمح لنا الالتزام بتسجيل هذه التغييرات.
الالتزامات هي لقطات من المشروع. يتم تسجيل كل التزام في الفرع الرئيسي للمستودع. يمكننا استدعاء الالتزامات أو إعادتها إلى الإصدار الأقدم. لن يتم استبدال التزامين مختلفين أبدًا لأن كل التزام له معرف الالتزام الخاص به. معرف الالتزام هذا هو رقم تشفير تم إنشاؤه بواسطة خوارزمية SHA (خوارزمية التجزئة الآمنة).

دعونا نرى أنواع الالتزامات المختلفة.
The git commit command
الأمر git الالتزام
سيقوم أمر الالتزام بتنفيذ التغييرات وإنشاء معرف الالتزام. سيفتح أمر التنفيذ بدون أي وسيطة محرر النص الافتراضي ويطلب رسالة الالتزام. يمكننا تحديد رسالتنا الالتزام في محرر النصوص هذا. سيتم تشغيله على النحو التالي:
$ git commit
